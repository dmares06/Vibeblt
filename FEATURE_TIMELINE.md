# Vibeblt Feature Timeline

This timeline keeps the free product focused on getting builders real feedback and helping visitors discover useful AI-built projects. Paid help and marketplace features are part of the long-term direction, but they should wait until the free discovery and feedback loop is active.

## Phase 1: Free Feedback + Discovery Foundation

Goal: make Vibeblt valuable before monetization by helping builders get useful feedback, attention, and early users.

- Turn project discussion into a real feature with authenticated comments, replies, and helpful reactions.
- Add project hearts or saves so visitors can signal interest and return to projects later.
- Make "needs feedback" a first-class browse path using the existing help tags.
- Add richer browse filters for category, project tags, help tags, newest, most liked, and most discussed.
- Add homepage sections for recently launched projects, projects needing feedback, most discussed projects, and AI agent tools.
- Add stronger project page calls to action: visit live project, give feedback, save project, and share listing.
- Add basic builder dashboard analytics for views, outbound project clicks, comments received, hearts, and saves.
- Improve empty states so visitors and builders always know what to do next.

## Phase 2: Community + Trust Loops

Goal: make projects, builders, and feedback feel credible enough that users return.

- Add fuller builder profiles with social links, bio, project count, follower count, and visible project history.
- Add launch/update history so visitors can see how actively a project is improving.
- Add quality guidelines for submissions so listings have useful screenshots, clear descriptions, and working links.
- Add verified live-link checks or visible "last checked" status for project URLs.
- Add curated collections such as "Built with AI agents," "Needs beta users," "Design critique wanted," and "Founder tools."
- Add notification emails for new comments, helpful reactions, project approval, and featured placement.
- Add weekly or monthly featured project digests once there are enough active submissions.

## Phase 3: Paid Help Marketplace

Goal: let builders pay for focused help only after there is evidence that users are requesting help repeatedly.

Not yet: do not add payments, escrow, paid listings, or paid help flows until the free feedback loop has enough real usage.

- Add "request paid help" as an optional action on projects with active help tags.
- Add helper profiles for people who can offer design critique, security review, performance work, growth help, app store submission help, and product feedback.
- Add a scoped help request form with budget, timeline, required skill, project context, and desired outcome.
- Add private messaging or an intake flow so builders and helpers can clarify scope before payment.
- Add paid offers with clear deliverables, price, turnaround time, revision limits, and cancellation terms.
- Add reviews for completed help requests so trust compounds over time.
- Add admin moderation for paid help categories, helper eligibility, disputes, refunds, and abuse reports.

## Phase 4: Pro + Marketplace Expansion

Goal: monetize only after Vibeblt has enough supply, demand, and repeat behavior.

- Add featured listings or promoted placement for builders who want more visibility.
- Add pro analytics with traffic sources, conversion to outbound clicks, save trends, and feedback trends.
- Add project brief posting so visitors can hire builders or helpers for new work.
- Add agency or team pages if multiple people begin using Vibeblt as a portfolio and lead-generation surface.
- Add saved collections for visitors who use Vibeblt as an inspiration and discovery tool.
- Add marketplace search for helpers by skill, availability, price range, and rating.

## Monetization Guardrails

- Keep project submission, browsing, public project pages, comments, basic saves, and basic analytics free while building the initial community.
- Do not require payment for feedback participation; paid help should be an upgrade path, not the core interaction.
- Do not launch paid help until there are enough repeated help-tag requests to show demand.
- Do not add complex marketplace policy before transactions exist, but define refund, dispute, and moderation requirements before accepting payments.
- Prioritize features that make builders share their projects and visitors return before adding revenue mechanics.
